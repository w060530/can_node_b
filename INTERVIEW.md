## 1. 项目概述（30 秒电梯演讲）

> "基于 STM32F103C8T6 和 FreeRTOS 的 CAN 总线双节点通信系统。两个节点通过 SN65HVD230 收发器在 500Kbps 速率下互发心跳帧和传感器数据帧，实现了 Bus-Off 自动恢复、序列号丢帧检测、硬件滤波器配置等工业级可靠性机制。项目使用 CubeMX 生成骨架 + VSCode + ARM GCC + CMake 开发，全程遵循 USER CODE 区域规范。"

## 3. 技术栈

| 层次 | 技术 | 说明 |
|------|------|------|
| **MCU** | STM32F103C8T6 | Cortex-M3, 72MHz, 64KB Flash, 20KB SRAM |
| **CAN 外设** | bxCAN (Basic Extended) | 14 个滤波器组, 3 级 RX FIFO, 3 个 TX 邮箱 |
| **CAN 收发器** | SN65HVD230 | 3.3V, 最高 1Mbps, RS 引脚可配置 |
| **RTOS** | FreeRTOS V10.3.1 (CMSIS-RTOS V2) | 抢占式调度, 3 个任务 + 1 个消息队列 |
| **HAL** | STM32F1 HAL | CAN/UART/GPIO 驱动 |
| **工具链** | ARM GCC 14.3 + CMake + Ninja | CubeMX 生成骨架，VSCode 编辑 |
| **调试器** | PW-Link (pyOCD) / ST-Link | 外部 GDB Server 模式 |
| **串口** | USART1, 115200 baud | printf 重定向调试输出 |

## 4. 关键设计决策（面试重点）

### 4.1 为什么 CanRxTask 优先级高于 CanTxTask？

| | 中断驱动 + 高优先级接收 | 低优先级接收 / 任务轮询 |
|---|---|---|
| **响应延迟** | μs 级 | ms 级 |
| **丢帧风险** | 低（ISR < 10μs） | 高！STM32F1 CAN FIFO 只有 3 级 |
| **CPU 占用** | 极低（仅在有帧时触发） | 持续轮询消耗 |

> **核心逻辑**：CAN 消息是异步到达的，不受本节点控制。500Kbps 下一帧约 130μs，若 osDelay(1ms) 轮询一次，期间最多来 7 帧，而硬件 FIFO 只有 3 级 → 必丢帧。接收任务永远不该等。

### 4.2 为什么用中断驱动而不是任务轮询？

- **中断**：ISR 中读 FIFO → 塞队列 → 返回（< 10μs），重活交给任务
- **轮询**：osDelay(1ms) 期间可来 7 帧，硬件 FIFO 3 级 → 丢 4 帧
- **标准做法**：汽车/工业 CAN 都是中断驱动

### 4.3 CAN 硬件 ACK ≠ 软件 ACK

- **硬件 ACK**：物理层 CRC 校验通过 + 至少一个节点在 ACK slot 拉低总线
- **软件 ACK**：应用层确认对方正确处理了消息（本项目用 Data[0]=0x04 的 ACK 帧）
- **类比**：硬件 ACK = "快递已签收"（可能是前台），软件 ACK = "邮件已读回执"（本人确认）

### 4.4 通信可靠性机制

| 机制 | 实现 | 效果 |
|------|------|------|
| **序列号** | 每帧 seq(0~255) 自增 | 检测丢帧、重复、乱序 |
| **超时检测** | 1s 未收到心跳 → 离线 | 及时发现断连 |
| **Bus-Off 恢复** | 直接读 CAN1->ESR 寄存器 | 节点独处后自动恢复 |
| **错误计数复位** | Error Passive/Warning 主动清零 TEC | 防累积恶化 |
| **滤波器配置** | STM32F1 必须显式配置 | 否则收不到任何帧 |

## 5. 踩坑记录（面试中的"problem solving"素材）

### 坑1：CAN 硬件滤波器未配置
- **现象**：回环自测时发送成功但收不到任何帧
- **原因**：STM32F1 CAN 硬件滤波器默认全关闭，需显式调用 `HAL_CAN_ConfigFilter()`
- **解决**：初始化时配置全通滤波器（掩码全 0 = 接受所有帧）
- **调试技巧**：在 `HAL_CAN_RxFifo0MsgPendingCallback` 打断点 → 不触发 = 问题在滤波层

### 坑2：Bus-Off 导致单节点启动后双节点不通
- **现象**：两个板子同时上电心跳正常；一个先上电一个后上电 → 不通
- **原因**：单节点在总线上发帧无 ACK → TEC 累积 → Bus-Off
- **解决**：直接读 `CAN1->ESR` 硬件寄存器检测 Bus-Off，主动恢复（停止→复位→重启→重配滤波器）
- **教训**：`HAL_CAN_GetError()` 返回缓存值不可靠，需读硬件寄存器

### 坑3：printf 在 ISR 中调用导致死机
- **原因**：`__io_putchar` 使用阻塞式 `HAL_UART_Transmit`，在 ISR 中调用会死等
- **解决**：所有 printf 只在任务中调用，ISR 只做"读 FIFO → 塞队列 → 返回"

### 坑4：STM32F1 CAN FIFO 只有 3 级深度
- **影响**：必须在 ~400μs 内读完 FIFO，否则溢出
- **解决**：CanRxTask 设最高优先级 + ISR 快进快出
- **波及**：所有 STM32F1 CAN 项目

### 坑5：ST-Link GDB Server 端口残留
- **现象**：Error 138 / 端口 61234 被占用
- **解决**：在 `tasks.json` 中增加 `preLaunchTask` 自动杀残留进程

## 6. 高频面试问题（自问自答）

### Q1: CAN 和 UART/SPI/I2C 的本质区别是什么？
> - **多主架构**：任意节点可在任意时刻发送（通过仲裁避免冲突）
> - **优先级仲裁**：ID 越小优先级越高（线与机制）
> - **错误处理**：5 种错误检测 + 错误计数器 + Bus-Off 机制
> - **帧结构**：有 ACK slot（硬件级确认），数据帧最多 8 字节
> - **应用场景**：汽车（动力/底盘/车身）、工业自动化

### Q2: 为什么 CAN 帧数据只有 8 字节？
> 历史设计折中：8 字节对实时控制足够（一个开关量/模拟量只需 1-2 字节），帧越短 → 实时性越好 → 更高优先级帧等待时间越短。CAN FD 已扩展到 64 字节。

### Q3: 标准帧 vs 扩展帧？
> - 标准帧：11-bit ID（0x000~0x7FF），2048 个 ID，IDE bit=0
> - 扩展帧：29-bit ID（~5.3 亿个），IDE bit=1
> - 本项目用标准帧，2 个节点只需 5 个 ID

### Q4: FreeRTOS 任务间通信用了什么？为什么不是全局变量？
> 消息队列（`osMessageQueue`）。CAN ISR 中不能阻塞 → 只能用 `osMessageQueuePut(timeout=0)`。全局变量不能阻塞等待，CanRxTask 会空转浪费 CPU。

### Q5: 为什么 STM32F103 而不是 STM32F4？
> F103 是 Cortex-M3，性价比高，CAN 外设（bxCAN）与 F4 兼容，学习成本低。F4 有更多滤波器组（28 vs 14），但本项目不需要。

### Q6: 如果总线上有 50 个节点，怎么设计 ID？
> 使用**分段编码**：优先级(3bit) + 节点地址(6bit) + 消息类型(2bit)，在 11-bit ID 内完成分区。

### Q7: 为什么不用 CANopen 协议栈？
> 本项目是 Demo 学习用途，手写协议层是为了理解每一层的工作原理。工业项目建议用 CANopen（CiA 301/302）或 J1939。

## 7. 代码亮点（面试时展示）

### 亮点1：直接读硬件寄存器做 Bus-Off 检测
```c
uint32_t esr = CAN1->ESR;  // 不是 HAL_CAN_GetError()！
if (esr & CAN_ESR_BOFF) { /* Bus-Off 恢复 */ }
```
**面试要点**：能解释为什么 HAL 封装的函数不可靠，敢用寄存器。

### 亮点2：ISR 快进快出 + 任务做重活
```c
// ISR: <10μs
void HAL_CAN_RxFifo0MsgPendingCallback(...) {
    HAL_CAN_GetRxMessage(...);   // 读硬件 FIFO
    osMessageQueuePut(..., 0);   // 塞队列（不阻塞！）
}
// 任务：协议解析、printf、状态更新
```
**面试要点**：理解 ISR 约束 + FreeRTOS API 在 ISR 中的使用限制。

### 亮点3：序列号丢帧检测
```c
if (seq != expected) {
    heartbeat_lost++;
    // 设 WARNING 标志 → LED 快闪
} else {
    // 连续正确 → 自动清除 WARNING
}
```
**面试要点**：体现工程思维——不仅仅是"能不能收到"，还包括"通信质量监控"。

### 亮点4：CubeMX USER CODE 区域规范
- 所有用户代码写在 `/* USER CODE BEGIN X */` 之间
- CubeMX 重新生成代码不会覆盖
- 面试要点：团队协作中代码可维护性的意识
